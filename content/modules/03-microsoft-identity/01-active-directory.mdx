# Active Directory Domain Services

Microsoft shipped Active Directory Domain Services in 1999 alongside Windows 2000 Server. Twenty-seven years later it still authenticates logins at the majority of enterprise organizations on earth. It is brittle, attack-surface-rich, 25-year-old software running mission-critical workloads — and you absolutely must understand it before anything else in this curriculum makes sense.

## What it is

AD DS is Microsoft's on-premises <Definition term="LDAP directory">A directory service built on the Lightweight Directory Access Protocol (RFC 4511). An LDAP directory is a hierarchical, attribute-based store optimized for read-heavy identity lookups. AD DS exposes LDAP on port 389 (or 636 for LDAPS) and stores every user, computer, group, and policy object in a replicated database called the NTDS.DIT.</Definition> combined with a Kerberos authentication service (RFC 4120), a DNS resolver, and a Group Policy engine. Those four components are deeply entangled — AD DS does not work without DNS, Kerberos does not work without AD DS, and Group Policy does not work without either.

The logical structure is a strict hierarchy: <Definition term="Forest">The outermost boundary of an Active Directory deployment. A forest contains one or more domains that share a single schema and a single global catalog. The forest is also the security boundary — a compromise of any domain in a forest must be treated as a compromise of all domains in the forest. This is the architectural fact that makes cross-forest trusts a strategic risk decision, not a technical one.</Definition> at the top, domains below it, and Organizational Units (OUs) inside domains. The schema — the master template that defines every object class and attribute — is forest-wide and replicated to every domain controller. This is a hard constraint: you cannot have a per-domain schema. If your forest has five domains and you extend the schema to support a custom application attribute, all five domains absorb that extension whether they want it or not.

## Why it exists

In 1999, the enterprise identity problem was straightforward in scope if not in implementation: how do you let one Windows domain account log into any machine in the building and access any file share it is authorized to reach? The answer was Kerberos — a ticket-granting protocol designed at MIT (RFC 1510, now superseded by RFC 4120) — layered on top of an LDAP directory that stored the accounts, and glued together by DNS so that clients could find domain controllers automatically via SRV records.

This was genuinely elegant for its era. The Kerberos model has a clean separation between authentication (proving who you are to the Key Distribution Center) and authorization (presenting a service ticket to a resource). Kerberos tickets are short-lived and cryptographically bound. Group Policy provided the first serious enterprise Configuration-as-Code: one XML-like document, linked to an OU, and every machine in that OU gets the security baseline applied automatically on next refresh. For 2000, this was remarkable.

The problem is that the world changed and AD DS did not. The protocol trust model that was elegant in a sealed office LAN is dangerous when laptops leave the building, when service accounts accumulate privileges over decades because nobody documented what they did, and when a single domain controller compromise gives an attacker Domain Admin over the entire forest.

<ProTip>
Microsoft's AD DS engineering team is in maintenance mode. Security patches ship; new capabilities do not. Every roadmap document Microsoft publishes steers you toward Entra ID for modern identity workloads. This does not mean you can ignore AD DS — most organizations will run it for another decade — but you should not be making architectural bets that depend on AD DS becoming more capable. Assume it will remain exactly as dangerous as it is today.
</ProTip>

## How it works under the hood

**The replication topology.** AD DS replicates via a multi-master model within a domain: any writable domain controller can accept a change, and the Knowledge Consistency Checker (KCC) automatically builds a replication topology between DCs using site and subnet objects you define. Sites and subnets are AD DS's model of your physical network — you tell AD DS which IP ranges belong to which site, and AD DS uses that information to route client logon requests to a nearby DC and to control when inter-site replication happens (you can throttle it to off-hours across slow WAN links). Get your sites-and-subnets configuration wrong and clients in a remote branch are authenticating across a 10 Mbps MPLS link instead of the DC sitting in the server room next to them.

**FSMO roles.** Five operations in AD DS cannot be multi-master safely, so Microsoft designates a single DC for each as a Flexible Single Master Operation role. The five roles, their scope, and what breaks when they go offline:

- **Schema Master** (forest-wide): the only DC that can write schema changes. Going offline means you cannot extend the schema; normal operations are unaffected.
- **Domain Naming Master** (forest-wide): must be online to add or remove a domain from the forest. Normal operations unaffected when offline.
- **PDC Emulator** (per-domain): the most operationally critical role. It processes password changes preferentially (so a just-changed password works before replication completes), synchronizes the domain's time hierarchy (Kerberos tickets fail if clocks drift more than 5 minutes — the PDC Emulator is the authoritative time source), and processes Group Policy changes. A PDC Emulator outage produces intermittent authentication failures, account lockout storms, and GPO application failures within minutes in a busy environment.
- **RID Master** (per-domain): allocates pools of Relative Identifiers to other DCs. A DC without RIDs cannot create new objects. Outage is low-impact until RID pools exhaust — then object creation halts.
- **Infrastructure Master** (per-domain): maintains cross-domain group membership references. Low operational impact except in multi-domain forests where it produces stale membership data if offline for extended periods.

The practical implication: the PDC Emulator is your highest-availability FSMO requirement. In any serious environment it lives on a virtualized DC with a very aggressive recovery SLA.

**The protocol triad.** AD DS exposes three protocols everything else depends on: LDAP (directory queries and object management), Kerberos (authentication ticket exchange), and DNS (DC and service location). Remove any one of them and AD DS stops functioning. This also means that anyone with network access to your domain controllers has three distinct protocol attack surfaces pointed at them — which is why DC placement behind tight firewall rules and the Tier 0 isolation model are non-negotiable in hardened environments.

**Group Policy as Configuration-as-Code.** A Group Policy Object (GPO) is a collection of settings — registry values, script execution, software installation, security policy, audit configuration — linked to a site, domain, or OU and applied to every computer and user object in scope at next refresh. GPOs process in LSDOU order: Local, Site, Domain, OU (innermost OU wins for conflicts). This inheritance model is powerful and merciless. A single misconfigured GPO linked to the domain root affects every machine in the domain.

**Group scoping — the AGDLP model.** AD DS gives you three group scopes — Global, Domain Local, and Universal — and the scope is not cosmetic. It dictates which objects a group can contain and where the group can be granted permissions. Microsoft's prescriptive way to wire those scopes together is **AGDLP**: **A**ccounts go into **G**lobal groups, Global groups nest into **D**omain **L**ocal groups, and the Domain Local group is the one you grant **P**ermissions on the resource. You put *who* (users, by role or department) into Global groups, you put *what they can do* (the permission on a share, an OU, a printer) onto Domain Local groups, and you connect the two by nesting. The payoff is that membership changes (a new hire, a transfer) happen once, in the Global group, and access flows through automatically — you never touch the resource ACL again.

<Definition term="AGDLP">Accounts → Global → Domain Local → Permissions. Microsoft's canonical role-based access pattern in a single-domain AD DS environment. Accounts are placed in Global groups (which represent roles or org units and only contain objects from their own domain). Global groups nest into Domain Local groups (which can pull members from any trusted domain but only grant access on resources in their own domain). The Domain Local group is what appears on the resource's ACL. This indirection is the on-prem ancestor of every RBAC model you will meet later in this curriculum.</Definition>

In a multi-domain forest, the chain grows one link to **AGUDLP**: **A**ccounts → **G**lobal → **U**niversal → **D**omain **L**ocal → **P**ermissions. Universal groups (whose membership replicates to the global catalog, so it is visible forest-wide) sit between the per-domain Global groups and the per-domain Domain Local groups, letting you collect role groups from several domains into one forest-wide bucket before granting access.

<ProTip>
Why not just dump user accounts straight onto resource ACLs, or grant permissions directly to Global groups? Because it does not scale and it does not survive multi-domain growth. Direct ACLs mean every access change is a resource-by-resource ticket, and Global groups cannot be granted permissions on resources in *other* domains. AGDLP/AGUDLP is the discipline that keeps "who has access" (Global/Universal) cleanly separated from "what access exists" (Domain Local) — and on the exam, when you see a question about which group scope holds members versus which one goes on the ACL, the answer almost always traces back to this model. Watch the trap: Universal group membership replicates to the global catalog, so churning a Universal group's members triggers forest-wide GC replication — which is exactly why you nest *Global* groups into Universal groups rather than individual users.
</ProTip>

**The attack surface.** BloodHound is an open-source graph analysis tool that ingests data from an AD DS environment and maps every privilege escalation path from an arbitrary user account to Domain Admin. It was released in 2016 and has been in continuous production use by red teams and attackers ever since. The existence of BloodHound is the most honest indictment of AD DS's security model available: the attack paths it finds are not exotic bugs — they are the logical consequence of permission inheritance, nested group membership, delegation rights, and Kerberoastable service accounts accumulated over years of normal operations.

The canonical attack chain: Kerberoasting extracts a service ticket encrypted with a service account's password hash from the KDC (no special privileges required — any domain user can request a service ticket for any SPN). The attacker takes the ticket offline and cracks the hash. If the service account has elevated privileges or a path to elevated privileges in BloodHound, the attacker follows that path to Domain Admin. Average time from initial Kerberoasting to Domain Admin in an unhardened environment on a red team engagement: under an hour.

## War story

<WarStory title="The 45-Minute Path to Domain Admin">
A regional community bank, 38 branches, about 800 employees. The AD DS forest was 22 years old — it predated the bank's last two core-banking platform migrations and had absorbed two acquisitions. Nobody alive at the bank had been there when the forest was built.

During a red team engagement, the initial foothold was a phished branch teller — a credential harvested from a fake password-reset email. The account had no special privileges. Standard domain user. The red team ran SharpHound (the BloodHound data collector) from the compromised workstation. Forty-five minutes after the initial foothold, they had Domain Admin.

The path BloodHound mapped: the teller's account was a member of a distribution list that had been granted "GenericWrite" on an OU containing IT helpdesk computers (someone had done this to allow automated printer installs years ago, never documented, never reviewed). GenericWrite on a computer object lets you set the `msDS-AllowedToActOnBehalfOfOtherIdentity` attribute, which enables Resource-Based Constrained Delegation (RBCD) abuse. From there, the red team impersonated a Domain Admin to the helpdesk workstation, harvested credentials from memory via the compromised session, and walked into the domain.

None of the components were bugs. Every permission was "by design" — it was just design from a decade ago that nobody had reviewed. There was no Tier 0 boundary. Domain controllers were on the same VLAN as member servers. There was no PAW. There were 47 accounts with Domain Admin rights, 12 of which appeared to belong to former employees.

The remediation project ran 14 months. The hardest part was not technical — it was organizational. Every quarter, someone wanted to re-add their admin account to Domain Admins "just to fix something real quick." The security team built an approval workflow using PIM (via Entra ID, for the hybrid-synced accounts) so that Domain Admin elevation required a ticket, a second approver, and a 4-hour window. The workflow was resented and fought. It held.

In a SaaS-first organization running Entra-only, there is no forest to compromise — but the conceptual equivalent is an Entra ID Global Administrator account with no PIM, no MFA, and a permanent active assignment. The attack path is different; the organizational pattern is identical. In a defense environment, air-gapped AD DS forests at classification boundaries mean there is no Entra Connect sync — the attack surface is contained to the network, which is why physical and logical access control to domain controllers is treated as a Tier 0 perimeter problem. In healthcare, Epic's integration with AD DS creates a common exception pattern: "Epic needs this service account to be a Domain Admin because the vendor said so." It is almost never true. Challenge it every time.
</WarStory>
