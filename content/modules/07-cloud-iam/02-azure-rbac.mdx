# Azure RBAC: The Resource Authorization Layer <SC300Badge />

In Module 3 you learned that Entra ID authenticates the user and issues the token. This section is about the layer that sits directly on top of that: once an identity is authenticated and holding a valid token, **what is it actually allowed to do to your Azure resources?** That question is answered by Azure role-based access control — Azure RBAC — the authorization system built into Azure Resource Manager.

Keep the two layers distinct in your head, because conflating them is one of the most common points of confusion on every Microsoft cloud exam and in real incident reviews. Entra ID issues the token (see Module 3 → Entra ID). Azure RBAC decides what that token can touch. They are different systems with different role catalogs, different scopes, and different management blades in the portal.

## What it is

Azure RBAC is the authorization layer of <Definition term="Azure Resource Manager (ARM)">The deployment and management plane for everything in Azure — the single control-plane API that every Azure resource operation flows through, whether you call it from the portal, the CLI, PowerShell, an SDK, or a REST client. Because every resource management request goes through ARM, ARM is the natural place to enforce authorization. Azure RBAC is the authorization model ARM consults on every control-plane request.</Definition>. When a principal asks ARM to do something — start a VM, read a storage blob, create a network security group — ARM checks Azure RBAC to decide whether that principal is allowed to perform that action at that scope.

<Definition term="Azure RBAC">The authorization system that controls access to Azure resources through Azure Resource Manager. It works by creating role assignments: a security principal is granted a role (a set of permitted actions) at a particular scope. It governs what an authenticated identity can DO to Azure resources — distinct from Entra ID directory roles, which govern administration of the Entra directory itself.</Definition>

The single most important distinction to lock in first is **Azure RBAC roles versus Entra ID (directory) roles.** They sound similar and people routinely mix them up:

- **Azure RBAC roles** control access to *Azure resources* through ARM. Examples: Contributor on a resource group, Storage Blob Data Reader on a storage account, Virtual Machine Contributor on a subscription. These answer "what can this identity do to my cloud infrastructure?"
- **Entra ID (directory) roles** control administration of *Entra ID itself* — the directory. Examples: Global Administrator, User Administrator, Privileged Role Administrator. These answer "what can this identity do inside the directory?" (create users, reset passwords, manage Conditional Access). Entra directory roles are covered in Module 3.

A user can be a Global Administrator in Entra (full control of the directory) and have *zero* access to any Azure resource, because that is a separate grant in a separate system. The reverse is also true: an Owner on a subscription does not, by virtue of that, get any directory privileges. The two systems intersect only in the sense that an Entra Global Administrator can, if they choose, elevate themselves to manage Azure RBAC across all subscriptions — but that is an explicit, auditable action, not an automatic inheritance.

## Why it exists

Before Azure RBAC, Azure used the classic subscription administrator model — Account Administrator, Service Administrator, Co-Administrator. That model had exactly one privilege level worth mentioning: effectively full control of the entire subscription. There was no way to say "this person can manage the VMs in the production resource group but cannot touch networking, and cannot see anything in the finance subscription at all." It was all-or-nothing per subscription, which is the opposite of least privilege.

Azure RBAC exists to make authorization **granular and scoped**. It lets you grant exactly the permissions an identity needs, at exactly the scope it needs them, and nothing more — the cloud-resource expression of the same least-privilege principle that NIST SP 800-53 AC-6 calls for. As organizations moved from one subscription with three administrators to dozens of subscriptions, hundreds of resource groups, and thousands of human and workload identities, the classic model became unworkable. Azure RBAC, layered on ARM and integrated with Entra ID as the identity source, is what made governed multi-subscription Azure possible.

## How it works under the hood

### The three elements of a role assignment

Every grant of access in Azure RBAC is a **role assignment**, and a role assignment is always the combination of three things:

> **security principal** + **role definition** + **scope**

Think of it as a sentence: *this principal can do these actions at this scope.*

1. **Security principal** — *who* gets the access. This can be a user, a group, a service principal (an application), or a managed identity. Assigning to groups rather than individual users is the strongly recommended practice; it keeps the assignment count down and makes joiner-mover-leaver changes a matter of group membership.
2. **Role definition** — *what* they can do. A role definition is a named collection of permissions, expressed as four arrays: `Actions` (allowed control-plane operations), `NotActions` (operations subtracted from Actions), `DataActions` (allowed data-plane operations, like reading the bytes of a blob), and `NotDataActions` (data-plane subtractions). Effective permissions for a role are `Actions` minus `NotActions`, and `DataActions` minus `NotDataActions`.
3. **Scope** — *where* the access applies.

<Definition term="role definition">A named collection of permissions in Azure RBAC, defined by four arrays: Actions (allowed management operations), NotActions (operations excluded from Actions), DataActions (allowed data-plane operations such as reading blob contents), and NotDataActions (data-plane exclusions). Built-in role definitions are managed by Microsoft; custom role definitions are authored by you.</Definition>

### The scope hierarchy — four levels, inherited downward

Scope is where Azure RBAC gets its power and its subtlety. There are four scope levels, from broadest to narrowest:

> **management group → subscription → resource group → resource**

<Definition term="management group">The top level of the Azure scope hierarchy, above subscriptions. A management group is a container you use to organize multiple subscriptions and apply governance — role assignments and Azure Policy — to all of them at once. A role assignment at a management group is inherited by every subscription, resource group, and resource beneath it. Management groups can be nested, and all of them roll up to a single tenant root management group.</Definition>

<Definition term="scope">The set of resources to which a role assignment applies. In Azure RBAC, scope is one of four levels — management group, subscription, resource group, or resource — and an assignment made at one level is inherited by everything below it. Granting at the narrowest scope that satisfies the requirement is the least-privilege practice.</Definition>

The critical rule: **assignments are inherited downward.** A role granted at a subscription applies to every resource group and resource inside that subscription. A role granted at a management group applies to every subscription under it. This is what lets you set broad organizational baselines high in the hierarchy and narrow, specific grants low in it.

When a principal has *multiple* role assignments that apply to a given resource — say, Reader at the subscription and Contributor on one resource group inside it — the effective permissions are the **union** of all applicable assignments. Azure RBAC is additive: the principal in that example has Contributor on resources in that one resource group (Reader plus Contributor unions to the broader set) and Reader everywhere else in the subscription. Two assignments never cancel each other out, and unioning two roles never magically produces a third role like Owner.

### How Azure RBAC evaluates a request — and how that differs from AWS

This is worth contrasting deliberately with the AWS policy-evaluation model from section 01, because the two clouds reason about access very differently and exams love to probe the difference.

- **Azure is additive for Allow.** There is no concept of "denied because you were not explicitly allowed by every guardrail." If any applicable role assignment grants the action, and no deny assignment blocks it, the action is allowed. Effective access is the union of all role assignments at and above the resource's scope.
- **Deny assignments override Allow.** Azure has a separate construct — the deny assignment — that *blocks* specific actions for specific principals at a scope, and a deny assignment wins over any role assignment that would otherwise allow the action.

<Definition term="deny assignment">An Azure RBAC construct that blocks specific actions for specified principals at a scope, overriding any role assignment that would otherwise allow them. Deny assignments are created and managed by Azure itself — for example through deployment stacks, Azure managed applications, and (historically) Azure Blueprints — to protect governed resources. (Azure Policy enforces configuration through its own *Deny* effect, which returns a 403 at request time and is a separate mechanism, not an RBAC deny assignment.) You cannot author a deny assignment directly the way you create a role assignment.</Definition>

So the Azure precedence is short: **deny assignment > role-assignment Allow > default no access.** Compare that to AWS, where the request starts at implicit deny, an explicit Deny in *any* policy wins, and then the action must be allowed by *each* applicable guardrail (SCP, resource policy, permission boundary, session policy, identity policy) before it succeeds. AWS layers multiple restrictive policy types that each must allow; Azure simply unions allows and lets a comparatively rare deny assignment override them.

<ProTip>
Hold the three cloud evaluation models side by side as you study. **AWS:** start denied, explicit deny always wins, then each guardrail (SCP, boundary, session, identity/resource policy) must allow — boundaries and SCPs only restrict, never grant. **Azure:** additive union of role-assignment allows, with deny assignments (created only by Azure, not by you) overriding. **GCP** (next section): additive union of allow policies, with deny policies evaluated first. All three let a deny override an allow, but only AWS makes you pass through a stack of restrictive policy types to get a yes.
</ProTip>

### Built-in roles versus custom roles

Azure ships a large catalog of **built-in roles** maintained by Microsoft. Four general-purpose ones cover most situations and are worth memorizing cold:

| Built-in role | What it grants | The catch |
| --- | --- | --- |
| **Owner** | Full access to manage all resources, **including the right to assign roles** (delegate access) | The only general role that can grant access to others |
| **Contributor** | Full access to create and manage all resources | **Cannot grant access to others** — no role-assignment rights |
| **Reader** | View all resources | Read-only; no changes |
| **User Access Administrator** | Manage user access (assign roles) | Can delegate access but **cannot manage the resources themselves** |

The Owner-versus-Contributor distinction is the classic exam item: both can fully manage resources, but only **Owner** can hand out access. User Access Administrator is the mirror image of Contributor — it can manage who has access but cannot touch the resources. Beyond these four, Microsoft publishes hundreds of fine-grained, service-specific built-in roles (for example, **Storage Blob Data Reader**, **Virtual Machine Contributor**, **Key Vault Secrets User**). Prefer the narrowest built-in role that fits before reaching for a custom one.

When no built-in role fits, you author a **custom role**: a JSON definition with the same `Actions`, `NotActions`, `DataActions`, and `NotDataActions` arrays as a built-in role, plus an `AssignableScopes` array declaring where the role is allowed to be assigned. Custom roles count against tenant limits — a tenant supports up to 5,000 custom roles, and a single subscription supports up to 4,000 role assignments (the standard limits at the time of writing; confirm current figures in Microsoft Learn before a sitting). The right answer when you start brushing against the role-assignment ceiling is almost always to assign roles to **groups** rather than individual principals, which collapses many assignments into one.

### Managed identities — the role-assignment angle <SC300Badge />

Module 3 introduced managed identities at the identity level: a **system-assigned** managed identity is tied one-to-one to a single Azure resource and shares that resource's lifecycle, while a **user-assigned** managed identity is a standalone object you can attach to many resources. Both let an Azure-hosted workload obtain an Entra token with no secret stored in code. (See Module 3 → Entra ID for the system-versus-user distinction and how the workload retrieves its token from the metadata endpoint.)

The Module 7 value-add is the authorization half of the story. A managed identity is just another **security principal**, so you grant it access to Azure resources the same way you grant access to a user or group: **you create a role assignment that gives the managed identity a role at a scope.** For example, you assign a virtual machine's system-assigned managed identity the **Storage Blob Data Reader** role on a specific storage account. From that point the VM's workload can read blobs from that account using a token the platform issues and rotates automatically — no connection string, no access key, no secret anywhere in code or configuration.

This is the heart of **secretless workload access** in Azure, and it is where the resource-authorization layer and the identity layer meet. Entra ID issues the workload a token that proves *who* it is; the Azure RBAC role assignment you create defines *what* that token can touch. Get the scope right — assign at the single storage account, not at the subscription — and you have a workload identity that follows least privilege by construction.

<ProTip>
When you grant a managed identity a role, assign it the *data-plane* role, not just a control-plane role, if the workload needs to read or write the actual data. A common stumbling block: an app's managed identity is given **Reader** or **Contributor** on a storage account but still gets "authorization failed" reading blob contents, because reading blob bytes is a `DataAction` covered by **Storage Blob Data Reader**, not by the management-plane Contributor role. Control-plane roles manage the resource; data-plane roles touch the data inside it.
</ProTip>

### Azure Policy — a different question entirely

Azure RBAC and **Azure Policy** are frequently confused because both are "governance," but they answer fundamentally different questions:

- **Azure RBAC** controls *who can perform which actions* on resources.
- **Azure Policy** controls *what resource configurations are allowed to exist.*

<Definition term="Azure Policy">A governance service that evaluates Azure resources against rules (policy definitions) and enforces or reports on configuration compliance — for example, deny creation of public IP addresses, require a specific tag, or audit untagged resources. It governs the configuration of resources, not who can act on them, with effects including Audit, Deny, Append, Modify, and DeployIfNotExists. Multiple policies are grouped into initiatives (policy sets).</Definition>

Azure Policy does not care *who* is acting — it cares about the *shape* of the resource. A policy with a **Deny** effect that blocks public blob access will stop even an **Owner** from creating a non-compliant storage account, because the policy evaluates the configuration of the request, not the requester's role. Policy effects include **Audit** (log non-compliance), **Deny** (block the operation), **Append** and **Modify** (alter the request), and **DeployIfNotExists** (deploy a remediation resource when something is missing). You bundle related policies into an **initiative** (a policy set) to manage them as a unit.

The cross-cloud parallel worth carrying forward: Azure Policy is the rough analog of AWS configuration guardrails (Config rules, and the configuration-shaping side of SCPs/RCPs), **not** the analog of AWS identity permissions. RBAC governs identity permissions; Policy governs resource configuration. Keep them in separate mental buckets.

### Azure Lighthouse — cross-tenant delegated management

One more piece matters if you ever manage Azure for someone other than your own organization — directly relevant to the consulting and MSP path. **Azure Lighthouse** enables **cross-tenant delegated resource management**: a service provider manages a customer's Azure resources from the provider's *own* Entra tenant, through **delegated role assignments** scoped to the customer's subscription or resource group, without ever creating guest accounts in the customer's tenant.

The customer's administrator onboards the provider once, choosing which built-in roles the provider's users and groups receive and at which scope. The provider's engineers then see and manage the delegated resources alongside their own from a single pane, using their home-tenant identities. Lighthouse keeps the authorization model clean: it is still ordinary Azure RBAC role assignments — they are simply projected across the tenant boundary — so everything you know about scope, built-in roles, and least privilege still applies. For an MSP managing dozens of customer tenants, this beats juggling dozens of guest accounts and dozens of sets of credentials.

## War story

<WarStory title="The Contributor Who Could Not Grant — and the Owner Sprawl It Caused">
A mid-size insurer was standing up its first governed Azure environment — three subscriptions (prod, non-prod, shared services) under a single management group, about 60 engineers. The platform team, trying to do the right thing, decided no one would be Owner; everyone doing real work would get **Contributor** at the appropriate resource group, and only two platform leads would hold **Owner** at the subscription for the rare role-assignment task.

It worked for about a week. Then the tickets started. A senior engineer needed to give a new contractor read access to one resource group so they could troubleshoot. As a Contributor, the engineer could create, delete, and reconfigure every resource in that group — but when they opened Access control (IAM) and clicked Add role assignment, the operation failed. Contributor cannot assign roles. Every small access change now funneled to the two overworked platform leads, who became a bottleneck. Within a month, under pressure, the leads started handing out **Owner** at the resource-group scope "just so people can manage their own access." Owner sprawl — the exact thing the design was meant to prevent — crept back in through the path of least resistance.

The fix was the role nobody had reached for: **User Access Administrator.** The platform team granted a small number of team leads User Access Administrator at their team's resource group, *in addition* to Contributor. That combination — Contributor (manage the resources) plus User Access Administrator (manage who has access to those resources) — gave team leads exactly what they needed to delegate read access to contractors without making them Owner of anything. The two roles unioned cleanly because Azure RBAC is additive. They then layered Entra PIM (Module 3) on top so even those elevated grants were eligible-not-active and time-bound.

The lesson generalizes across every environment. In a bank, the same mistake produces over-privileged Owners on the subscription that holds the core-banking workload — a finding every examiner writes up. In a government tenant, where separation of duties is mandated, the Contributor-versus-Owner-versus-User-Access-Administrator distinction is not a convenience, it is a control. The takeaway: do not reach for Owner to solve a delegation problem. Owner is full control *plus* the right to grant access; if all someone needs is to grant access, that is precisely what User Access Administrator is for. Match the built-in role to the actual need, and let Azure RBAC's additive model compose the rest.
</WarStory>
