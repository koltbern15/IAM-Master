### 01-pam-fundamentals — "Privileged Access Management: The Keys to the Kingdom"
**Word target:** ~2,900. **SC-300:** no badge, no `sc300` flag.

**Topics / study material (subtopic outline + sources to cite):**

- **What PAM is and the privileged-account taxonomy.** Define privileged access as access that exceeds standard user rights and can alter system/security state. Taxonomy to teach explicitly:
  - *Local admin accounts* (per-machine), *domain admin / Enterprise Admin / built-in Administrator* (AD), *service accounts* (non-interactive, run apps/services), *application/embedded credentials* (hardcoded secrets, connection strings), *cloud root / privileged cloud roles* (AWS root, Entra Global Admin, GCP Org Admin), *emergency/break-glass accounts*, *third-party/vendor privileged accounts*, *DevOps/CI-CD machine identities*.
  - Cite: **NIST SP 800-53 Rev 5** control family **AC (Access Control)** — specifically **AC-2 (Account Management)**, **AC-6 (Least Privilege)**, **AC-6(2)/AC-6(5)** privileged account restrictions; **AC-5 (Separation of Duties)**. [VERIFY the exact control enhancement numbers against the 800-53 Rev 5 catalog before publishing.]
  - Cite: **Gartner** defines PAM as the four "PASM/PEDM/secrets/CIEM" pillars — Privileged Account & Session Management (PASM), Privilege Elevation & Delegation Management (PEDM), secrets management, and cloud infrastructure entitlement management (CIEM). [VERIFY this is Gartner's current PAM market segmentation wording.]
- **Why privileged accounts are the prime target.** Map to the attack lifecycle: credential theft → lateral movement → privilege escalation → domain dominance. Reference **MITRE ATT&CK** tactics TA0006 (Credential Access) and TA0004 (Privilege Escalation). Note Verizon DBIR consistently attributes a large share of breaches to credential abuse [VERIFY the specific year/percentage before quoting a number — do not invent a stat].
- **The vault (secrets-at-rest).** Define a privileged credential vault: encrypted store, role-based retrieval, full audit, automatic credential rotation. Contrast a *vault* (stores and rotates the secret) with a *session broker* (brokers a connection without revealing the secret). This "secrets vs sessions" distinction is the conceptual spine of the whole module.
- **Session isolation / Privileged Session Management (PSM).** Define PSM: the admin connects to a jump/proxy host that opens the real session to the target, so the credential is never exposed to the endpoint and the session is recorded (keystroke + video). Define *session recording*, *session monitoring*, *live session termination*, *command filtering*.
- **Credential injection.** Define: PSM injects the secret directly into the target protocol (RDP/SSH/web) without the operator ever seeing or typing it. This is what makes session isolation meaningfully better than just "knowing the password."
- **JIT (Just-in-Time) access and standing privilege.** Define *standing privilege* (24/7 permanent rights) vs *just-in-time* (rights granted only for a bounded window on request). Define **least-standing-privilege / zero-standing-privilege (ZSP)** — the goal state where no account holds privilege at rest. Cite the JIT concept as defined by CyberArk/BeyondTrust/Delinea PAM literature and reflected in **NIST SP 800-207 (Zero Trust Architecture)** per-session, per-request access. [VERIFY 800-207 wording maps cleanly to JIT.]
- **JEA (Just Enough Administration).** Define Microsoft **JEA** as PowerShell-based constrained endpoints (role capability files + session configuration files) that grant only specific cmdlets/parameters via a virtual privileged account, leaving the operator a standard user. Cite **Microsoft Learn — Just Enough Administration**. Distinguish JEA (scope of what you can do) from JIT (when you can do it) — they compose.
- **Least privilege & privilege elevation (PEDM).** Define PEDM / endpoint privilege management: remove local admin from endpoints, elevate specific applications on demand with policy. Tie to NIST 800-53 AC-6.
- **The PAM control loop.** Discover → onboard → vault → rotate → broker session → record → review. (Sets up section 06.)
- **Banking framing.** FFIEC and PCI-DSS both require control over privileged/admin access; PCI-DSS 4.0 Requirement 7 (least privilege) and Requirement 8 (identification/authentication) are the relevant hooks. [VERIFY PCI requirement numbers — Req 7 = restrict access by business need-to-know; Req 8 = identify users and authenticate access. Confirm against PCI-DSS v4.0.]

### 02-cyberark — "CyberArk: The PAM Market Leader"
**Word target:** ~3,000. **SC-300:** no badge, no `sc300` flag. **NEEDS the new `<CyberArkArchDiagram />`.**

**Topics / study material:**

- **CyberArk's place in the market.** The long-standing PAM leader; flagship is the **Privileged Access Manager (Self-Hosted)** suite (formerly "Privileged Access Security / PAS"), plus the SaaS **Privilege Cloud**. Cite **CyberArk Docs** (docs.cyberark.com). [VERIFY current product naming: "Privileged Access Manager - Self-Hosted" and "Privilege Cloud" are the current SKUs as of 2024-2025.]
- **The Digital Vault (EPV / Enterprise Password Vault).** The hardened, encrypted core. Teach:
  - *Vault server* runs on a hardened Windows host, often air-gapped, with its own firewall (the Vault hardening locks down everything but the proprietary **Vault protocol** on TCP 1858). [VERIFY port 1858 is the Vault's listening port.]
  - *Layered encryption* — each password object encrypted with its own key; the **Server Key** required to open the vault at startup (can be stored on an HSM). Cite CyberArk Vault encryption hierarchy docs. [VERIFY the "Server Key / Storage / object keys" hierarchy wording.]
  - *Safes* — logical containers for credential objects with their own RBAC (Safe members + authorizations).
- **PVWA (Password Vault Web Access).** The web UI / primary user and admin interface; where users request, retrieve, and (with dual control) get approved access. Runs on IIS. Cite CyberArk PVWA docs.
- **CPM (Central Policy Manager).** The engine that performs **automatic credential rotation / reconciliation** against managed targets per the **Master Policy** and platform settings. Teach *change*, *verify*, *reconcile* operations and *platforms* (the per-target-type plugin defining how to rotate). Cite CyberArk CPM docs.
- **PSM (Privileged Session Manager) & PSM for SSH.** The session-isolation/jump-server tier — RDP/SSH/web sessions proxied, recorded, with **credential injection** so the secret never reaches the endpoint. PSM for SSH (formerly PSM-SSHP) handles Unix/Linux. Cite CyberArk PSM docs.
- **PTA (Privileged Threat Analytics).** Behavioral analytics that detects anomalous privileged activity (e.g. suspicious vault retrieval, Golden Ticket indicators) and can trigger automatic CPM rotation. Cite CyberArk PTA docs. [VERIFY PTA is still the current component name vs being folded into the broader Identity Security Platform.]
- **The Master Policy.** Central rule set governing dual control, exclusive/one-time passwords, session recording requirements, and check-in/check-out. Cite CyberArk Master Policy docs.
- **AAM / Conjur — application secrets (the app-to-app / non-human side).**
  - **Credential Providers (CP)** and **Central Credential Provider (CCP)** / **Application Access Manager (AAM)** eliminate hardcoded credentials in application code/config — the app fetches the secret at runtime authenticated by characteristics (path, hash, OS user). Cite CyberArk AAM/Credential Provider docs.
  - **Conjur** (Conjur Enterprise / open-source Conjur) — the DevOps/cloud-native secrets manager for CI/CD, containers, and microservices, with policy-as-code and machine identity. Cite CyberArk Conjur docs. Note its overlap/competition with HashiCorp Vault (section 04 cross-link).
- **Reference flow (drives the diagram).** Admin authenticates to PVWA → policy check / dual-control approval → PSM brokers an isolated, recorded session with CPM-rotated credentials injected → CPM rotates the credential on check-in → PTA watches for anomalies. The vault never hands the cleartext to the operator.
- **Banking relevance.** CyberArk is the dominant PAM in financial services; emphasize audit/recording for examiner evidence and the operational weight of the Vault as a single hardened crown-jewel.

### 04-hashicorp-vault — "HashiCorp Vault: Secrets as Code"
**Word target:** ~2,800. **SC-300:** no badge, no flag.

**Topics / study material:**

- **What HashiCorp Vault is.** A secrets-management and data-protection platform built for dynamic, cloud-native/DevOps environments — API-first, identity-brokered, with secrets-as-code. Contrast its developer/automation posture against the human-privileged-session posture of CyberArk/BeyondTrust/Delinea. Cite **HashiCorp Vault docs** (developer.hashicorp.com/vault). [VERIFY current product framing now under IBM ownership post-2024 acquisition — note the acquisition but keep product facts vendor-doc-sourced.]
- **Seal / unseal & the barrier.** Vault starts *sealed*; the **barrier** encrypts all storage. Unsealing reconstructs the master key via **Shamir's Secret Sharing** (key shares / threshold) or **auto-unseal** (cloud KMS/HSM Transit). Define *root token*, *unseal keys*, *seal/unseal*. Cite Vault seal concepts docs.
- **Secrets engines.** Pluggable backends mounted at paths:
  - **KV v2** — versioned static key-value store; teach the *v1 vs v2* difference (v2 adds versioning + soft delete + metadata) and the path quirk (`secret/data/<path>` for the API vs `secret/<path>` in the CLI). Cite Vault KV v2 docs.
  - **Dynamic secrets** — the killer feature: Vault *generates* short-lived, on-demand credentials (e.g. **database** engine mints a unique DB user per request; **AWS** engine mints temporary IAM creds; **PKI** engine issues short-lived certs). The credential never pre-exists and auto-expires with its lease. Cite Vault dynamic-secrets / database / AWS / PKI engine docs.
  - **Transit** — "encryption as a service": Vault encrypts/decrypts data without storing it (cryptographic offload). Cite Vault Transit docs.
- **Auth methods.** How clients prove identity to *get* a token: **AppRole** (role_id + secret_id, for apps/CI), **Kubernetes** (service-account JWT), cloud IAM (**AWS/Azure/GCP** auth), **LDAP/OIDC/userpass**, **TLS certs**, **tokens**. Cite Vault auth-methods docs. Distinguish auth methods (who you are) from secrets engines (what you get).
- **Leases, renewal, and revocation.** Every dynamic secret and token has a **lease** with a TTL; clients renew before expiry; Vault auto-revokes on lease end, and an operator can revoke a lease (or a whole prefix) to instantly kill leaked credentials. This is the dynamic-secrets safety property. Cite Vault lease docs.
- **Policies (ACL policies).** HCL/JSON path-based capability grants (`create`, `read`, `update`, `delete`, `list`, `sudo`, `deny`) attached to tokens via auth roles. Cite Vault policies docs. Tie to least privilege.
- **Identity & namespaces (Enterprise).** Entities/aliases unify identities across auth methods; **namespaces** (Enterprise) give multi-tenant isolation. [VERIFY namespaces are Enterprise-only.]
- **Why dynamic > static.** Reframe: static long-lived secrets are the attack surface PAM tries to vault and rotate; Vault's dynamic secrets shrink the surface to near-zero TTL. This is the conceptual payoff and a great cross-link back to 01's "secrets vs sessions."

### 06-pam-program-design — "Designing a PAM Program"
**Word target:** ~2,700. **SC-300:** no badge, no flag.

**Topics / study material:**

- **The PAM program lifecycle.** Discover → assess/prioritize → onboard → enforce (vault/rotate/session) → monitor → review/recertify → mature. Anchor to a maturity model and to NIST.
- **Discovery.** Find every privileged account, local admin, service account, SSH key, hardcoded secret, and SPN across AD, servers, network gear, databases, SaaS, and cloud. Tools: vendor discovery engines (CyberArk Accounts Discovery, BeyondTrust Smart Rules, Delinea Discovery) + native (`Get-LocalGroupMember`, AD queries). Emphasize that you cannot protect what you have not discovered.
- **Onboarding workflow.** Bring discovered accounts under management: assign to safes/folders with RBAC, set rotation platform/policy, enable session brokering, decide check-out vs always-rotate. Sequence by risk (domain admins and crown-jewel service accounts first).
- **Break-glass / emergency access.** Design the offline, sealed, monitored emergency path *for the PAM system itself and for the directory*: at least two break-glass accounts, cloud-only/excluded-from-CA (cross-link Module 3 §02 break-glass card), credentials split/sealed (dual control), alerting on any use, periodic test. Cite NIST 800-53 **CP/AC** and Microsoft's emergency-access-account guidance. [VERIFY MS Learn "emergency access accounts" guidance citation.]
- **Rollout scoping & sequencing.** Phase by population and risk (Tier 0 / domain admins → Tier 1 servers → Tier 2 workstations; cf. Microsoft's **Enterprise Access Model**, successor to the legacy tiered model). Pilot → expand → enforce, with change control and a back-out plan. Cite **Microsoft Learn — Enterprise Access Model / privileged access strategy**. [VERIFY "Enterprise Access Model" is the current name superseding "Tier 0/1/2 administrative tiering."]
- **Least-standing-privilege as the end state.** Drive toward JIT/ZSP; measure standing-privilege reduction over time.
- **Service accounts & secrets.** The hardest part of any real program — discover, classify (interactive vs non-interactive), convert to gMSA/managed identity where possible, vault and rotate the rest, eliminate hardcoded secrets via CCP/Conjur/Vault.
- **Metrics & audit evidence.** % privileged accounts under management, % with auto-rotation, # standing admin accounts, session-recording coverage, time-to-onboard, recertification completion. These are the numbers an FFIEC/PCI examiner asks for. Map to PCI-DSS Req 7/8 and SOX 404 ITGC access controls. [VERIFY framework mappings.]
- **Common failure modes.** Boiling-the-ocean (too broad a first phase), breaking a critical service account rotation, no break-glass test, treating PAM as a tool deployment instead of a program. (Tie back to the war stories across the module.)

[VERIFY the specific maturity framework wording you cite.]

## Flashcards (drafted — author inline as `<Flashcard front="..." back="..." />`)

> All cards below avoid embedded double quotes (extraction requirement). Verify every factual claim against the cited vendor/NIST doc before publishing; `[VERIFY]` tags flag the ones most likely to need a product-version check.

### 01-pam-fundamentals (9)
1. What is privileged access? -> Any access that exceeds standard user rights and can change system, security, or identity state — local/domain admin, service accounts, cloud root/admin roles, embedded application credentials, and break-glass accounts. NIST 800-53 AC-6 frames it as the access governed by least privilege.
2. What is the difference between a credential vault and a session broker? -> A vault stores and rotates the secret and hands it out under policy (secrets-at-rest); a session broker opens the privileged connection on the operator's behalf so the operator never sees the secret (session isolation). PAM does both — vaulting and brokering.
3. What is Privileged Session Management (PSM)? -> A proxy/jump tier that opens the real session to a target on the admin's behalf, injects the credential without revealing it, and records the session (keystroke and video) for audit and live monitoring/termination.
4. What is credential injection in PSM? -> The session manager inserts the vaulted secret directly into the target protocol (RDP/SSH/web) at connect time, so the operator authenticates without ever seeing or typing the password — defeating credential theft from the endpoint.
5. Define standing privilege vs just-in-time (JIT) access. -> Standing privilege is permanent 24/7 rights held at rest; JIT grants the rights only for a bounded window on request and revokes them after. JIT shrinks the attack window from permanent to minutes/hours.
6. What is zero-standing-privilege (ZSP) / least-standing-privilege? -> The target state where no account holds privileged rights at rest; every privileged action requires a just-in-time, time-bound grant. It is the strongest expression of least privilege (NIST 800-53 AC-6).
7. What is Just Enough Administration (JEA)? -> A Microsoft PowerShell capability that exposes constrained endpoints (role-capability + session-configuration files) granting only specific cmdlets/parameters via a temporary virtual privileged account, leaving the operator a standard user.
8. How do JIT and JEA compose? -> JEA limits WHAT a privileged operator can do (scoped cmdlets); JIT limits WHEN they can do it (time-bound activation). Used together they enforce least privilege across both scope and time.
9. Which NIST 800-53 controls anchor privileged-account management? -> AC-2 (Account Management), AC-5 (Separation of Duties), and AC-6 with its privileged-account enhancements (Least Privilege). [VERIFY the exact AC-6 enhancement numbers against 800-53 Rev 5.]

### 02-cyberark (10)
1. Name the four core components of the CyberArk self-hosted PAM suite. -> The Digital Vault (EPV) for encrypted storage, PVWA (Password Vault Web Access) for the web UI, CPM (Central Policy Manager) for automatic credential rotation, and PSM (Privileged Session Manager) for isolated, recorded sessions.
2. What is the CyberArk Digital Vault and why is it hardened? -> The encrypted, layered-key core store for all privileged secrets. It runs on a hardened, often isolated host exposing only the proprietary Vault protocol, because compromise of the vault would expose every managed credential. [VERIFY Vault listens on TCP 1858.]
3. What is a Safe in CyberArk? -> A logical container inside the vault that groups credential objects and carries its own role-based access (Safe members and authorizations) — the unit at which you delegate who can use which secrets.
4. What does the CPM (Central Policy Manager) do? -> It automatically changes, verifies, and reconciles managed credentials on their targets according to the Master Policy and platform settings — the engine behind automatic rotation.
5. What does PSM (Privileged Session Manager) provide beyond vaulting? -> Session isolation: it proxies RDP/SSH/web sessions, injects credentials so the secret never reaches the endpoint, and records sessions for audit, live monitoring, and termination.
6. What is the CyberArk Master Policy? -> The central rule set governing privileged-access behavior — dual control/approval, exclusive or one-time passwords, mandatory session recording, and check-in/check-out — applied across safes.
7. What is the Server Key in CyberArk and why does it matter? -> The key required to open the vault at startup; without it the vault stays sealed. It is often stored on an HSM, making it the root of trust for the entire vault. [VERIFY Server Key role wording.]
8. What is CyberArk PTA (Privileged Threat Analytics)? -> A behavioral analytics layer that detects anomalous privileged activity (suspicious retrievals, Kerberos attack indicators) and can trigger automatic credential rotation in response. [VERIFY PTA is still a distinct current component.]
9. What problem do CyberArk Credential Providers / CCP (AAM) solve? -> They remove hardcoded credentials from application code and config — the app fetches its secret at runtime, authenticated by attributes like path, application hash, and OS user, instead of storing a static password.
10. What is Conjur and how does it differ from the classic CyberArk vault? -> Conjur is CyberArk's DevOps/cloud-native secrets manager with policy-as-code and machine identity for CI/CD, containers, and microservices — built for automated, high-velocity app-to-app secrets rather than human privileged sessions.

### 03-beyondtrust-delinea (8)
1. What is BeyondTrust Password Safe? -> BeyondTrust's PASM product: privileged credential discovery, vaulting, automatic rotation, and session management/recording, with Smart Rules driving automated account onboarding. [VERIFY Smart Rules naming.]
2. What is BeyondTrust Endpoint Privilege Management (EPM)? -> A PEDM product that removes local admin rights from Windows/Mac/Unix-Linux endpoints and elevates approved applications on demand by policy — least privilege at the endpoint. [VERIFY current EPM naming vs PowerBroker/Defendpoint lineage.]
3. What is BeyondTrust Privileged Remote Access (PRA) used for? -> Giving vendors and third parties brokered, recorded privileged access to internal systems without a VPN or shared credentials — a major control for third-party access in banking.
4. Delinea was formed from which two companies, and when? -> From the 2021 merger of Thycotic and Centrify. Old product names like Thycotic Secret Server and Centrify now sit under the Delinea brand.
5. What is Delinea Secret Server? -> Delinea's flagship vault: secrets storage with RBAC, automatic rotation via Remote Password Changing (RPC), session launching and recording, check-out/exclusive access, and credential validity checks via Heartbeat. [VERIFY RPC and Heartbeat naming.]
6. What is the check-out / exclusive access model in a PAM vault? -> A user reserves a credential exclusively for a window; others cannot use it until check-in, and the secret is typically rotated on check-in — guaranteeing accountability for one operator at a time.
7. In one line, how do CyberArk, BeyondTrust, and Delinea generally differ in positioning? -> CyberArk targets the largest, most-regulated estates (deep, heavy); BeyondTrust leads on a unified endpoint-privilege + remote-access story; Delinea Secret Server is known for faster, lighter time-to-value. [VERIFY as general market positioning, not a benchmark.]
8. What is the difference between PASM and PEDM? -> PASM (Privileged Account/Session Management) vaults, rotates, and brokers/records sessions for shared privileged accounts; PEDM (Privilege Elevation and Delegation Management) removes standing local admin and elevates specific actions/apps on the endpoint by policy.

### 04-hashicorp-vault (8)
1. What is a HashiCorp Vault dynamic secret? -> A credential Vault generates on demand for a short lease rather than storing a pre-existing one — for example a unique database user per request — which auto-expires and is auto-revoked at lease end, shrinking the credential's exposure to near zero.
2. What is a secrets engine in Vault? -> A pluggable backend mounted at a path that stores or generates secrets — KV (static key-value), database/AWS/PKI (dynamic), Transit (encryption-as-a-service) — each with its own API and behavior.
3. What is a Vault lease and why does it matter? -> Every dynamic secret and token carries a lease with a TTL; clients renew before expiry and Vault auto-revokes at the end. An operator can revoke a lease (or whole prefix) to instantly kill leaked credentials.
4. What is the difference between a Vault auth method and a secrets engine? -> An auth method (AppRole, Kubernetes, AWS/Azure/GCP IAM, LDAP, OIDC, TLS) proves WHO a client is and issues a token; a secrets engine determines WHAT secret that token can obtain. Auth = identity, engine = access.
5. What is the AppRole auth method used for? -> Machine/automation authentication: an application presents a role_id plus a secret_id to obtain a Vault token, avoiding human credentials in CI/CD and app pipelines.
6. What is the seal/unseal process in Vault? -> Vault starts sealed with its storage barrier encrypted; unsealing reconstructs the master key from a Shamir threshold of key shares (or via cloud-KMS/HSM auto-unseal) before any secret can be read.
7. What is the difference between KV v1 and KV v2 in Vault? -> KV v2 adds secret versioning, soft delete, and metadata, and uses a different API path (secret/data/<path>); KV v1 is a simple unversioned key-value store. [VERIFY the v2 data-path detail.]
8. What does the Vault Transit engine do? -> Encryption as a service: Vault encrypts, decrypts, signs, and rotates keys for application data without ever storing the data itself, so apps offload cryptography without holding key material.

### 05-cloud-native-pam (9)
1. In Entra PIM, what is the difference between an eligible and an active role assignment? -> Eligible means the user may activate the role just-in-time (with MFA, justification, and optional approval) and holds no privilege until they do; active means the user holds the role now. The ideal is eligible-only for privileged roles.
2. Which Entra ID license tier is required for Privileged Identity Management? -> Entra ID P2. PIM, Identity Protection, access reviews, and entitlement management all sit in the P2 tier. [VERIFY against Module 3 to stay consistent.]
3. What does AWS Systems Manager Session Manager provide? -> Brokered, fully logged shell/RDP/port-forward access to instances without open inbound ports, bastions, or SSH keys — AWS-native privileged session isolation, with logs to CloudWatch/S3.
4. What is AWS STS AssumeRole and why is it the cloud JIT pattern? -> Security Token Service issues short-lived, scoped credentials when a principal assumes a role, replacing long-lived access keys — the AWS-native way to grant temporary, just-in-time privileged access.
5. What does GCP OS Login do? -> It binds Linux SSH access to Google IAM identities and roles (compute.osLogin / compute.osAdminLogin) instead of per-host authorized_keys files, centralizing SSH key and access lifecycle in IAM. [VERIFY the two role names.]
6. What is GCP Identity-Aware Proxy (IAP) used for in a PAM context? -> Brokered TCP/SSH access to instances without public IPs or a VPN, with access governed by IAM — the GCP-native equivalent of a bastion-less, identity-gated session path.
7. When do you choose native cloud PAM over a dedicated PAM platform? -> Native (PIM, Session Manager, OS Login) fits single-cloud, infra-as-code shops; dedicated PAM (CyberArk/BeyondTrust/Delinea) wins for multi-cloud plus on-prem unification with consistent recording and human-session governance. They are commonly complementary.
8. How does PIM produce audit evidence for compliance? -> Every eligible-role activation, approval, justification, and expiration is logged, giving SOX/FFIEC examiners a first-class record that privileged access was time-bound and controlled.
9. What is the cloud reframing of privileged access? -> In cloud, privileged access is often role activation plus ephemeral credentials rather than a vaulted static password — JIT role elevation (PIM/STS/IAM Conditions) replaces standing keys.

### 06-pam-program-design (6)
1. Why is discovery the first step of a PAM program? -> You cannot vault, rotate, or monitor what you have not found; discovery enumerates every privileged account, local admin, service account, SSH key, and hardcoded secret across AD, servers, databases, SaaS, and cloud before any control is applied.
2. How should you sequence PAM onboarding? -> By risk: highest-impact accounts first — domain/Global admins (Tier 0) and crown-jewel service accounts — then servers, then workstations, piloting and expanding rather than boiling the ocean.
3. What are the design requirements for break-glass accounts? -> At least two, cloud-only and excluded from Conditional Access, with long sealed credentials split under dual control, alerting on any use, and periodic testing — so an outage or misconfigured policy never fully locks out admins.
4. What is Microsoft's Enterprise Access Model? -> The current privileged-access tiering guidance (successor to the legacy Tier 0/1/2 model) that isolates control planes by sensitivity so a lower-tier compromise cannot escalate to control-plane (Tier 0) assets. [VERIFY current naming.]
5. Why are service accounts the hardest part of a PAM program? -> They are numerous, often undocumented, frequently have hardcoded or non-rotating credentials, and break applications if rotated carelessly; the fix is discover, classify, convert to gMSA/managed identity where possible, and vault/rotate the rest with eliminate-the-hardcoded-secret patterns.
6. Name four PAM program metrics an examiner would ask for. -> Percentage of privileged accounts under management, percentage with automatic rotation, number of remaining standing-admin accounts, and session-recording coverage — plus recertification completion and time-to-onboard. [VERIFY mapping to PCI-DSS Req 7/8 and SOX 404 ITGC.]
