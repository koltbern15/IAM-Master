# Module 12 — Hands-On Labs (Phase 2) — Authoring Plan

> **Goal:** Author the 13-lab capstone module (`content/modules/12-labs/`) that turns every prior module's theory into command-driven, do-it-yourself labs, so Kolton builds, breaks, and fixes real IAM systems end-to-end.

**Architecture note:** No new engineering. This is pure MDX authoring plus the standard three-touch wiring (modules.json + sections.ts + content-loader.ts). Every lab is one `.mdx` file (clean prev/next, no grouping). The module is command-heavy: it leans on the existing `<PowerShellBlock>` component for the bulk of its body and reuses the Module 6 recipe philosophy. `newDiagram = ""` — no new diagram component is required.

**Prereqs / conventions (read before authoring a single file):**
- **Dependency ordering — author this module LAST.** The labs reference and operationalize Modules 3 (AD/Entra/Hybrid), 4 (PAM/CyberArk/PIM), 5 (IGA/SailPoint/Okta/Entra Governance), 6 (PowerShell), and 8 (identity threat detection). Several labs are the "now actually do it" counterpart to a concept section in those modules. Authoring before they exist means cross-links point at 404s and the prose can't say "as you saw in Module 5 §02." **Build order: 2A (Mod 4) → 2B (Mod 5) → 2C (Mod 8) → 2D (this module).** This is stated in the roadmap §3 Workstream 2.
- **Two-file sync (a unit test enforces it).** Adding sections requires updating BOTH `content/modules.json` (`12-labs` → `sections: [...]` slug array) AND `lib/sections.ts` (`SECTION_TITLES` map entry for every slug + add SC-300-aligned slugs to `SC300_SECTIONS`). A drift test fails if a slug exists in one and not the other. Change them together.
- **Content-loader registration.** Each new MDX file needs (a) its key added to `AUTHORED_SECTIONS` AND `ALL_KNOWN_SECTIONS` in `lib/content-loader.ts`, and (b) a matching `case` in `loadAuthoredComponent()` with a static `import('@/content/modules/12-labs/<slug>.mdx')`. This is the only app-code touch.
- **No YAML frontmatter.** Existing sections (e.g. `content/modules/06-powershell/02-cookbook.mdx:1`) open with a bare MDX comment `{/* ... */}` then the `<HoloPanel>` then `# H1` — there is NO gray-matter frontmatter block; `content-loader.ts` does not parse one. Match that exactly. Section title comes from `SECTION_TITLES`, NOT from any frontmatter.
- **Interactive content is INLINE MDX**, extracted at runtime by `lib/content-index.ts`. There are no JSON sidecars. Author `<Flashcard>`, `<Quiz>`, `<Definition>`, `<WarStory>`, `<ProTip>`, `<PowerShellBlock>`, `<HoloPanel>` inline.
- **`<Flashcard>` extractor constraints (`lib/content-index.ts:90`):** the regex is `/<Flashcard\b([^>]*?)\/>/` with `front="([^"]*)"` and `back="([^"]*)"`. Therefore each card MUST be **self-closing**, and the `front`/`back` attribute values **must not contain a literal double-quote** (use single quotes inside, or `&quot;` which the decoder converts back). No newlines inside attribute values.
- **`<Quiz>` shape (`components/content/Quiz.tsx:9`):** `<Quiz question={{ id: "...", prompt: "...", options: [...4...], correctIndex: N, explanation: "..." }} />`. The search extractor (`lib/content-index.ts:133`) keys off `prompt: "..."`, so prompt must be a double-quoted string.
- **`<PowerShellBlock>` shape:** `<PowerShellBlock title="...">{\`...multi-line command...\`}</PowerShellBlock>` — the child is a template literal. Inside it, a backtick line-continuation must be escaped as `` \` `` (see `02-cookbook.mdx:44`). Dollar-brace `${...}` PowerShell subexpressions render fine inside the JSX template literal because MDX treats the whole thing as a JS template string — but `$()` and bare `$var` are safe; only `${` would be interpolated by JS, so write PowerShell `$(...)` not `${...}` unless you intend JS interpolation. **When a command uses `$($var.Prop)` it is fine; avoid `${var}` PowerShell syntax — rewrite as `$($var)`.** [VERIFY] this escaping on first author + build.
- **House style:** top `<HoloPanel label="COVERS">` block mapping to SC-300 / Security+ / CISSP where relevant, then prose with `<Definition>`/`<WarStory>`/`<ProTip>`, embedded `<PowerShellBlock>` for every command step, light `<Flashcard>` recall, and a reflective `<Quiz>` only where it fits. Labs run **~1,500–2,500 words** each (lighter than the ~2,700-word concept sections because commands carry the weight).
- **ACCURACY IS NON-NEGOTIABLE.** Cite the real MS Learn page / vendor doc / RFC / NIST SP for every procedure. Never invent a cmdlet, flag, or product behavior. Draft uncertain syntax and append `[VERIFY]` for the author to confirm against live docs before publishing. Several CyberArk / SailPoint / Okta steps are drafted from public docs and marked `[VERIFY]` because they depend on tenant/version specifics.

## Module-level framing (author once, in Lab 01's intro and the module overview summary)

Module 12 is the **capstone**. Every lab follows the same skeleton so the module feels like one author wrote one lab manual:

1. `<HoloPanel label="COVERS">` — cert mappings + "Builds on: Module X §YZ".
2. `# H1` (matches `SECTION_TITLES`).
3. **Objective** — one paragraph: what you will have working when done, and why it matters for a banking IAM role.
4. **Prerequisites** — environment, licenses/SKUs, prior lab dependency, accounts/permissions needed. Always name the prior module section it builds on.
5. **Lab steps** — numbered `### Step N:` headings, each with a short rationale + a `<PowerShellBlock>` (or vendor-portal callout where no cmdlet exists).
6. **Validation** — how to prove it worked (a verification command + expected output shape).
7. **Teardown** — how to cleanly remove what you built (cost control + repeatability). Critical for the Azure/Entra labs (PIM, CA) and the cloud-cost labs.
8. **What this proves** — a reflection paragraph tying the muscle memory back to the concept and to the job/cert.
9. Recall: ~2 `<Flashcard>` on the key takeaway. Quizzes: mostly N/A — add a tiny reflective `<Quiz>` only where a genuine decision point exists (labs 03, 06, 10, 11 below).

**Lab safety banner (author as a `<ProTip>` near the top of Lab 01 and reference thereafter):** every destructive command is shown with `-WhatIf` first; never run bulk writes against a production tenant; use a dedicated **dev/test Entra tenant** (Microsoft 365 Developer Program / pay-as-you-go) and a throwaway lab VM. Cite MS Learn "Set up a Microsoft 365 developer subscription." [VERIFY] developer-program availability terms as of 2026.

### `01-build-an-ad-lab` — Build an Active Directory Lab From Scratch
**Word target:** ~2,200. **SC-300:** No (foundational; AD DS is on-prem). `<SC300Badge />` = no, `sc300` flag = false.

**Topics / study material (the substance to teach):**
- **Objective:** stand up a single-forest, single-domain AD DS lab (`corp.lab`) on a Windows Server VM, populate it with OUs/users/groups, so every later on-prem lab (LAPS, JML, Kerberoasting, stale-cleanup) has a sandbox.
- **Builds on:** Module 3 §01 Active Directory; Module 6 §01 PowerShell fundamentals.
- **Prereqs:** Hyper-V or VirtualBox; Windows Server 2022/2025 eval ISO (cite MS Eval Center); 4 GB+ RAM for the DC; a static IP plan. Cite MS Learn "Install Active Directory Domain Services (Level 100)."
- **Steps:**
  1. Provision the VM, set a static IP + rename the host (`Rename-Computer`). [VERIFY] eval ISO link.
  2. Install the AD DS role: `Install-WindowsFeature -Name AD-Domain-Services -IncludeManagementTools`. (MS Learn: `Install-WindowsFeature`.)
  3. Promote to a new forest: `Install-ADDSForest -DomainName "corp.lab" -InstallDns -SafeModeAdministratorPassword (Read-Host -AsSecureString)`. (MS Learn: `Install-ADDSForest`.) Reboots automatically.
  4. Create an OU structure that mirrors a real org (`Install-ADDSForest` done → `New-ADOrganizationalUnit`): `OU=Corp` with child `Users`, `Groups`, `Workstations`, `Servers`, `ServiceAccounts`, `Admins`.
  5. Bulk-create test users from a small CSV with `Import-Csv | New-ADUser` (mirror the Module 6 bulk-create pattern, but on-prem cmdlets). Set `-Path` to the right OU.
  6. Create AGDLP groups (`New-ADGroup -GroupScope Global` / `-GroupScope DomainLocal`) and nest them — ties to Module 3 §01 AGDLP coverage.
  7. Create a deliberately weak service account (SPN-registered, weak password) — this seeds Lab 10 (Kerberoasting). Note the linkage explicitly.
- **Validation:** `Get-ADDomain`, `Get-ADUser -Filter * -SearchBase "OU=Users,OU=Corp,DC=corp,DC=lab"`, `Get-ADOrganizationalUnit -Filter *`. Expected counts.
- **Teardown:** snapshot the VM after build (revert is the teardown); demote with `Uninstall-ADDSDomainController` only if dismantling.
- **What this proves:** you can build the substrate the rest of the curriculum manipulates; you understand forest promotion, OU design, and AGDLP from the inside, not from a slide.

**Components to embed:** `<HoloPanel>`, lab-safety `<ProTip>`, 5–6 `<PowerShellBlock>`, `<Definition term="domain controller (DC)">`, `<WarStory>` (a flat-OU "everyone in CN=Users" environment that made delegation impossible), 2 `<Flashcard>`.

**SC-300 alignment:** Not SC-300. No badge, `sc300:false`.
